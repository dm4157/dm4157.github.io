可以。通过观察发现，不同项目只有配置文件的内容是必须的，而bean的配置是基本是一致的。那么把bean配置抽到JAR包里可以么？可以，但会带来一个问题，引入缓存框架的项目都要提供ehcache memcached redis三者的配置文件，不论是否使用。就算解决这个问题，在Spring容器里也会存在三种缓存的Bean，程序员是肯定不会喜欢这种方案的（代码洁癖） 。如果能判断项目在使用哪种缓存并且自动配置相关的bean，那就完美了。
根据项目引入了何种缓存JAR，我们可以做出此类判断，接着只要能自动配置就好了。不可能只有我想到，所以Spring Boot已经实现了这种方案，参见spring-boot-autoconfigure。
Spring Boot 自动配置实现了如果类加载路径出现了相关JAR，就引入对应的Bean配置，用户只需要提供必要的配置文件（如链接地址）即可。至此只要引入spring-boot-autoconfigure就能解决公司缓存框架面临的问题。

@Configuration

可以注意到@Configuration是被@Component修饰的，所以可以被component-scan发现。

避免扫描

Spring 会自动扫描base-package下的Java类，如果发现带有@Component(包括用其修饰的注解@Repository @Service @Controller @RestController @Configuration等)的类，将会放入Spring容器中管理，成为众多Bean里的一员。
component-scan还包括两个子标签<context:include-filter>和<context:exclude-filter>,注意两个不是必须的，但如果同时出现，要保证include-filter出现在前面，并且如果范围重合，重合部分被后续的exclude-filter清掉。两个子标签都具有type和expression两个属性：

  • type
    • annotation 拥有指定注解的类
    • aspectj AspectJ语法
    • assignable 指定class或interface的全名
    • regex 表达式，可以写类似org.spring.*,org.spring包下的类都被包括
    • custom Spring自定义类型，并没有研究过
  • expression 就是表达式了，可以写全限定类名、符合规范的正则表达等

好了，至此如何避免自动配置类被扫描到，技能GET。

加入扫描

在避免了基础扫描后，我们要增加何时的扫描以引入自动配置的类。这里需要用到接口DeferredImportSelector。

突破口：DeferredImportSelector

这是一个典型的“标识型”接口，没有任何需要实现的方法。来看下官方注释：

A variation of ImportSelector that runs after all @Configuration beans have been processed. This type of selector can be particularly useful when the selected imports are @Conditional.
Implementations can also extend the Ordered interface or use the Order annotation to indicate a precedence against other DeferredImportSelectors.

大意是这个接口的实现类将会在@Configuration的Bean处理后执行。并且DeferredImportSelector是专门用来选择@Conditional修饰的Bean。我们也可以用@Ordered决定多个DeferredImportSelector的执行顺序。
我们的突破口就在这里， 让我们更详尽的了解一下接口ImportSelector的作用。

在@Configuration引入的类中，一般来说根据 importingClassMetadata 的信息，选择并返回类的符合要求的类名数组。执行顺序是在@Configuration引入的类之后。接口ImportSelector的作用类似于@Import.不过一个ImportSelector通常应该实现任意Aware接口：

  • EnvironmentAware
  • BeanFactoryAware
  • BeanClassLoaderAware
  • ResourceLoaderAware
